# Почему текущая реализация MIL не является полной

Этот документ нужен для честного объяснения границ проекта на защите и в ВКР.

## Короткий ответ

В проекте реализован **MIL-based reference engine** с **validated subset**, но не полный MIL-HDBK-217F end-to-end.

Причина не в том, что код «не работает», а в том, что handbook намного шире текущего MVP:
- в нём много отдельных разделов по семействам компонентов;
- для части семейств нужны точные таблицы и разветвлённые правила выбора строки;
- для Part Stress требуются специальные коэффициенты и ветвления по технологии, качеству, температуре, нагрузке, корпусу, среде эксплуатации;
- для system-level нужны отдельные board/interconnection additions.

## Что в проекте реализовано честно

### 1. Parts Count — validated subset
Для части семейств `parts_count` действительно опирается на транскрибированный subset Appendix A:
- resistors;
- capacitors;
- diodes / BJT subset;
- inductors / transformers;
- relays;
- connectors / sockets;
- quartz crystals;
- fuses;
- ограниченный subset microcircuits.

Если строка попадает в этот subset, в `assumptions` явно пишется, что использован `MIL-HDBK-217F Appendix A (validated subset)`.

### 2. Parts Count fallback остаётся в проекте
Если для строки нет validated MIL row, проект **не притворяется**, что всё равно считает exact MIL.
Вместо этого используется legacy fallback, и это также помечается в `assumptions.reference_method`.

### 3. Section 16 / 17 не скрыты
Board-level MIL reference уже вынесен в отдельный прозрачный слой и показывается в summary.
Он пока **не включается автоматически** в итоговый total FIT, чтобы не подмешивать в расчёт неявные предположения.
Section 17 connection rows тоже не считаются автоматически, потому что стандартный BOM не содержит counts соединений.

## Что делает текущую реализацию неполной

### 1. Part Stress сейчас не равен exact handbook formulas
`part_stress.py` сейчас использует упрощённую инженерную модель:
- базовые FIT по семействам;
- температурные множители;
- stress-ratio множители;
- family/subfamily modifiers.

Это полезный рабочий расчёт для MVP, но это **не полный цифровой перенос** формул разделов 5–23 handbook.

### 2. Microcircuits покрыты только частично
Для ИС в validated subset есть только ограниченные rows:
- часть microcontroller / memory paths;
- только явные MIL-like quality aliases;
- без полного переноса всей логики linear / logic / VHSIC / custom-screening mapping.

Поэтому весь IC-контур нельзя честно называть полностью реализованным по MIL.

### 3. Quality mapping сокращённый
В handbook quality mapping богаче и детальнее.
В проекте используется консервативное сведение внутренних quality aliases к меньшему набору MIL-like классов.
Это честный инженерный компромисс, но не полный handbook coverage.

### 4. Не все семейства перенесены в validated subset
Даже в parts-count режиме закрыты не все возможные строки handbook.
Часть семейств до сих пор уходит в legacy MVP constants.

### 5. Telcordia / EPFL не встроены как параллельные production-контуры
В ТЗ упомянуты дополнительные источники данных и альтернативные методы.
В текущем release основной расчётный контур — MIL-based.
Отдельного полнофункционального Telcordia plugin сейчас нет.

## Как это правильно формулировать

### Нельзя говорить
- «У меня полностью реализован MIL-HDBK-217F».
- «Все коэффициенты handbook перенесены один в один».
- «Part Stress полностью совпадает с оригинальными формулами MIL».

### Правильно говорить
- «В проекте реализован MIL-based reference engine».
- «Для parts-count режима доступен validated subset MIL Appendix A».
- «Part Stress реализован как инженерный расчётный контур MVP и пока не покрывает весь handbook целиком».
- «Board/interconnection references показываются отдельно и пока не включаются автоматически в total FIT».

## Практический смысл для защиты

Это не ослабляет проект, если подавать его честно.
Наоборот, сильная позиция такая:
- продукт реально обрабатывает BOM;
- reference-контур трассируемый;
- границы применимости обозначены явно;
- fallback и manual review сделаны прозрачно;
- система не имитирует точность там, где её нет.
